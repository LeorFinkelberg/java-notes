Ключевое слово `synchronized` можно применять к блоку или методу. Оно означает, что перед входом в блок или метод поток должен установить соответствующую блокировку [[Список литературы#^eb6c16]]<c. 177>

Например, представьте метод, который снимает деньги с банковской карты
```java
// В любой момент времени только один поток может пытаться снять средства с этого счета
public synchronized boolean withDraw(int amount) {
  if (balance >= amount) {
    balance = balance - amount; // balance -= amount;
	return true;
  }
  
  return false;
}
```

Метод должен установить блокировку на экземпляре объекта (а для статических синхронизированных методов -- на объекте `Class`). В случае блока программист должен указать, какой объект нужно заблокировать.

==NB! В любой момент времени ТОЛЬКО ОДИН ПОТОК может обращаться к синхронизированным блокам или методам объекта; если к ним попытаются обратиться другие потоки, JVM их приостановит. Это происходит вне зависимости от того, пытается ли другой поток войти в тот же или другой синхронизированный блок того же объекта [[Список литературы#^eb6c16]]<c. 177>

Основые моменты:
- Блокировать можно только объекты, но не примитивы,
- Блокировка массива объектов не блокирует отдельные объекты,
- Синхронизированный метод можно рассматривать как эквивалент блока `synchronized (this) {...}`, который охватывает весь метод (но учтите, что в байт-коде эти конструкции представляются по-разному),
- Метод, объявленный как `static synchronized`, устанавливает блокировку для объекта `Class`, потому что не существует экземпляра, который можно было бы блокировать,
- `synchronized` не входит в сигнатуру метода и поэтому не может присутствовать в объявлении метода интерфейса,
- Несинхронизированные методы не обращают внимание на состояние блокировок и могут продолжить работу в то же время, когда выполняются синхронизированные методы,
- Блокировки Java реентерабельны.

Почему используется ключевое слово `synchronized`. После того, как синхронизированный метод (или блок) завершается, любые изменения, внесенные в блокируемый объект, актуализируются в основной памяти до снятия блокировки. После того как при входе в синхронизированный блок устанавливается блокировка, любые изменения в блокируемом объекте читаются _из_ ==основной памяти==, так что поток с блокировкой синхронизируется с представлением объекта в основной памяти, прежде чем код в заблокированной секции начнет выполняться [[Список литературы#^eb6c16]]<c. 185>

==Изменения в объекте распространяются между потоками через основную память==[[Список литературы#^eb6c16]]<c. 186>

Одна из потенциальных проблем _неизменяемых объектов_ заключается в том, что их фабричным методам методам может передаваться множество параметров. Чтобы решить эту проблему, можно воспользоваться паттерном "Строитель". В нем совмещаются две конструкции: статический внутренний класс, который реализует обобщенный интерфейс строителя, и частный конструктор для самого неизменяемого класса.

Строитель представляет собой _обобщенный интерфейс верхнего уровня_, который обычно определяется так
```java
public interface Builder<T> {
  T build();
}
```

Это так называемый SAM-тип (Single Abstract Method, то есть "единичный абстрактный метод"), и формально он может быть целевым типом для лямбда-выражений. Однако задача строителя -- создавать неизменяемые экземпляры: вся его суть заключается в том, чтобы конструировать состояние, а не представлять функцию или обратный вызов. Это значит, что хотя строитель можно использовать как функциональный интерфейс, на практике это абсолютно бессмысленно. 

NB! По этой причине мы ==НЕ помечаем интерфейс строителя аннотацией `@FunctionalInterface`!== [[Список литературы#^eb6c16]]<c. 196>

Корректное использование API строителя таково: один поток использует строитель, чтобы накопить все необходимое состояние, затем создать неизменяемый объект, а уже этот объект могут совместно использовать другие потоки. 

Конечная ссылка может указывать на объект, который содержит неконечные поля. Иногда это называется _поверхностной неизменяемостью_ (shallow immutability) [[Список литературы#^eb6c16]]<c. 197>.

Например
```java
final var numbers = new LinkedList<Integer>();
```

В этой инструкции ссылка `numbers` и целочисленные объекты, которые содержатся в списке, ==неизменяемы==. Однако сам объект списка остается _изменяемым_, потому что в список можно добавлять целочисленные объекты, удалять и заменять их.

==NB! Если объект не может измениться, то нет никаких проблем с тем, чтобы изменения были видны всем потокам== [[Список литературы#^eb6c16]]<c. 200>
